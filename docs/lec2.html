2.3 Slides

2.3.1 Git mental model

  • Working directory (your files) ‚Üí git add ‚Üí staging ‚Üí git commit ‚Üí local history
  • Remote: GitHub hosts a copy. git push publishes commits; git pull brings others‚Äô changes.
  • Branch: a movable pointer to a chain of commits. Default is main. Create feature branches for each change.

In Git, a branch is essentially just a movable pointer to a commit.

2.4 1. The simple definition

  • A branch has a name (e.g., main, feature/login).
  • That name points to a specific commit in your repository.
  • As you make new commits on that branch, the pointer moves forward.

2.5 2. Visual example

Let‚Äôs say your repo looks like this:

A --- B --- C   ‚Üê main

Here:

  • main is the branch name.
  • It points to commit C.

If you make a new branch:

git branch feature

Now you have:

A --- B --- C   ‚Üê main, feature

If you checkout feature and make a commit:

A --- B --- C   ‚Üê main
             \
              D   ‚Üê feature
  • feature moves forward to D (new commit).
  • main stays at C.

2.6 3. HEAD and active branch

  • HEAD is your current position ‚Äî it points to the branch you‚Äôre working on.
  • When you commit, Git moves that branch forward.

2.7 4. Why branches matter

  • Let you work on new features, bug fixes, or experiments without touching the main codebase.
  • Cheap to create and delete ‚Äî Git branching is just updating a tiny file.
  • Enable parallel development.

2.8 5. Branches vs tags

  • Branch ‚Üí moves as you commit.
  • Tag ‚Üí fixed pointer to a commit (used for marking releases).

üí° Inside .git/refs/heads/, each branch is just a plain text file storing a commit hash.

In git checkout -b, the -b means ‚Äúcreate a new branch‚Äù before checking it out.

2.8.1 Without -b

git checkout branchname
  • Switches to an existing branch.
  • Fails if the branch does not exist.

2.8.2 With -b

git checkout -b branchname
  • Tells Git: ‚Äúmake a branch called branchname pointing to the current commit, and then switch to it.‚Äù
  • Fails if the branch already exists.

2.8.3 Example

If you‚Äôre on main:

git checkout -b feature-x

Steps Git takes:

  1. Create a new branch pointer feature-x ‚Üí same commit as main.
  2. Move HEAD to feature-x (you‚Äôre now ‚Äúon‚Äù that branch).

üí° In newer Git versions, the same idea is expressed with:

git switch -c feature-x   # -c means create

-b in checkout and -c in switch both mean create.

2.8.4 Branch & PR etiquette

  • One feature/change per branch (small, reviewable diffs).

  • Commit messages: imperative mood, short subject line (‚â§ 72 chars), details in body if needed:

    • feat: add git-lfs tracking for parquet
    • docs: add README section on setup
    • chore: ignore raw data directory

  • PR description: what/why, testing notes, checklist. Tag your teammate for review.

2.8.5 .gitignore must‚Äëhaves

  • Secrets: .env, API keys (never commit).
  • Large/derived artifacts: raw/interim data, logs, cache, compiled assets.
  • Notebooks‚Äô checkpoints: .ipynb_checkpoints/.
  • OS/editor cruft: .DS_Store (for Mac), Thumbs.db (for Windows), .vscode/.

2.8.6 Git‚ÄëLFS

  • Git‚ÄëLFS = Large File Storage. Keeps pointers in Git; binaries in LFS storage.
  • Track only what‚Äôs necessary to version (e.g., small processed Parquet samples, posters/PDFs, small models).
  • Do not LFS huge raw data you can re‚Äëdownload (make get-data).
  • Quotas apply on Git hosting‚Äîbe selective.

2.8.7 Safe pushes from Colab

  • Use a fine‚Äëgrained PAT limited to a single repo with Contents: Read/Write + Pull requests: Read/Write.
  • Enter token via getpass (not stored). Push using a temporary URL (token not saved in git config).
  • After push, clear cell output.

2.9.15 10) Open a Pull Request

The name ‚Äúpull request‚Äù can be confusing at first ‚Äî it sounds like you are ‚Äúpushing‚Äù your code, but really you‚Äôre asking someone else to pull it.

2.10 üìå Origin of the term

  • The phrase comes from distributed version control (like Git before GitHub‚Äôs UI popularized it).

  • If you had changes in your branch/repo and wanted them in the upstream project, you‚Äôd contact the maintainer and say:

    ‚ÄúPlease pull these changes from my branch into yours.‚Äù

  • So a pull request is literally a request for someone else to pull your commits.

2.11 üìå How it works (e.g., on GitHub, GitLab, Bitbucket)

  1. You push your branch to your fork or to the remote repository.
  2. You open a pull request against the target branch (usually main or develop).
  3. The repository maintainers review your code.
  4. If accepted, they ‚Äúpull‚Äù your commits into their branch (though under the hood it‚Äôs often implemented as a merge or rebase).

2.12 üìå Contrast with ‚Äúpush‚Äù

  • Push: You directly upload commits to a remote branch you have permission to write to.
  • Pull request: You don‚Äôt merge directly ‚Äî instead, you ask maintainers to pull your changes, review them, and integrate them.

‚úÖ Summary: It‚Äôs called a pull request because you‚Äôre not pushing your changes into the target branch; you‚Äôre asking the project owner/maintainer to pull your branch into theirs.

  • Recommended (web UI): Navigate to your repo on GitHub ‚Üí Compare & pull request ‚Üí base: main, compare: setup/git-lfs. Fill title/description, tag your partner, and create the PR.

  • Optional (API): open a PR programmatically from Colab:

# OPTIONAL: Create PR via GitHub API (requires token again)

import requests, json

token = getpass("GitHub token (again, not stored): ")

           "Accept": "application/vnd.github+json"}

2.14.2 Part B ‚Äî Add a large‚Äëfile guard (simple Python script)

Create a small tool that fails if files > 10 MB are found and aren‚Äôt tracked by LFS. This will be used manually for now (automation later in CI).

# tools/guard_large_files.py
import os, subprocess, sys

LIMIT_MB = 10
ROOT = os.getcwd()

def lfs_tracked_paths():
    try:
        out = subprocess.check_output(["git", "lfs", "ls-files"], text=True)
        tracked = set()
        for line in out.strip().splitlines():
            # line format: "<oid> <path>"
            p = line.split(None, 1)[-1].strip()
            tracked.add(os.path.normpath(p))
        return tracked
    except Exception:
        return set()

def humanize(bytes_):
    return f"{bytes_/(1024*1024):.2f} MB"

lfs_set = lfs_tracked_paths()
bad = []
for dirpath, dirnames, filenames in os.walk(ROOT):
    # skip .git directory
    if ".git" in dirpath.split(os.sep):
        continue
    for fn in filenames:
        path = os.path.normpath(os.path.join(dirpath, fn))
        try:
            size = os.path.getsize(path)
        except FileNotFoundError:
            continue
        if size >= LIMIT_MB * 1024 * 1024:
            rel = os.path.relpath(path, ROOT)
            if rel not in lfs_set:
                bad.append((rel, size))

if bad:
    print("ERROR: Large non-LFS files found:")
    for rel, size in bad:
        print(f" - {rel} ({humanize(size)})")
    sys.exit(1)
else:
    print("OK: No large non-LFS files detected.")

Add a Makefile target to run it:

# Create/append Makefile target
from pathlib import Path
text = "\n\nguard:\n\tpython tools/guard_large_files.py\n"
p = Path("Makefile")
p.write_text(p.read_text() + text if p.exists() else text)
print("Added 'guard' target to Makefile")

Run locally/Colab:
